# FC Wallet（未来公民钱包）设计框架方案

## 1. 文档目的

本文档用于统一 `FC Wallet` 的产品定位、设计框架、信息架构与阶段性落地建议，供产品、设计、研发及相关合作团队内部对齐使用。

它回答的不是“做一个什么样的钱包”这么简单的问题，而是：

- `FC Wallet` 在整个 `Future Citizen` 体系中到底扮演什么角色
- 为什么它不能被设计成传统加密钱包
- 它的功能应如何分层
- 它的页面结构和交互框架应如何建立
- 第一阶段应优先设计和落地哪些能力

## 2. 产品定位

### 2.1 产品定义

`FC Wallet` 不是单纯的资产钱包，也不是孤立的 DID 工具，而是 `Future Citizen` 体系面向用户的主权级数字入口。

它的产品本质应被定义为：

`一个集身份、凭证、支付、安全恢复、组织协作、可信通信与治理参与于一体的主权身份与操作平台。`

### 2.2 在体系中的角色

结合目前已学习的蓝图、链路资料与 PRD，可明确 `FC Wallet` 在体系中的角色如下：

- 它是公民进入 `Future Citizen System` 的第一入口
- 它是 `FCDID` 的承载容器与交互界面
- 它是资产和支付能力的执行载体
- 它是 `VC / ZKP / DIDComm` 等可信交互机制的用户侧入口
- 它是数字企业 `E-Company` 的操作工作台
- 它是未来线下 `NFC` 物理卡与现实场景的连接器
- 它是连接 `Bureau / Authority / Chain` 三层体系的用户前台

### 2.3 一句话定位建议

对内建议定义：

`FC Wallet 是 Future Citizen 生态的统一应用壳层。`

对外建议定义：

`FC Wallet 是面向未来公民的可信数字网关，连接身份、资产、凭证、组织与安全交互。`

## 3. 设计总原则

为了保证后续所有界面与交互方向统一，建议整个产品设计遵循以下原则。

### 3.1 身份优先，不是资产优先

传统钱包首页通常以余额、价格、转账为核心，但 `FC Wallet` 不适合这样设计。

在本产品中，用户首先需要知道的是：

- 我是谁
- 我拥有哪些身份与凭证
- 我当前具备哪些可信能力
- 我能发起哪些被系统认可的交互

资产是重要模块，但不应成为唯一中心。

### 3.2 信任必须可见

`Future Citizen` 的核心支柱是 `Identity / Trust / Governance`。因此，界面中不能只显示“功能”，还必须显示“可信状态”。

建议可视化呈现：

- 身份绑定状态
- 国家级 Digital ID 联动状态
- 凭证签发方可信性
- 组织身份真实性
- 官方消息真实性
- 安全恢复是否已完成配置

### 3.3 把链的复杂度藏起来

用户不应被迫理解底层区块链的复杂逻辑，例如：

- Gas 如何计算
- FCC 余额是否足够
- 路由如何选择
- 分片和共识是什么
- 合约调用如何执行

这些内容应默认被系统抽象掉，仅在高级详情或透明披露页面中按需展示。

### 3.4 隐私不是设置项，而是核心价值

`ZKP` 与“选择性披露”不是技术注脚，而是产品差异化核心。

设计上应让用户直观感受到：

- 我可以证明某件事
- 我不必暴露全部原始信息
- 每一次披露都可见、可控、可审查

### 3.5 个人与组织上下文必须严格区分

由于产品同时承载个人公民与数字企业场景，因此必须明确区分用户此刻是在：

- 以个人身份操作
- 以组织身份操作

尤其在以下行为中必须强提醒：

- 付款
- 签署
- 多签审批
- 凭证出示
- 治理投票

### 3.6 整体风格应体现机构感与未来感

产品视觉语气建议偏向：

- 严肃
- 可信
- 未来感
- 机构级秩序感
- 高效而非投机

应避免将界面做成典型“交易所风格”或“纯 Web3 炫技风格”。

## 4. 功能框架总览

建议将 `FC Wallet` 的功能划分为八大层。

### 4.1 身份层

核心能力：

- `FCDID` 创建与管理
- `did:fc` 身份展示
- `FNS` 名称解析与管理
- 国家 `Digital ID` 绑定
- 身份等级或可信状态呈现

设计目标：

- 让用户把钱包理解为“我的数字身份门户”

### 4.2 凭证与隐私层

核心能力：

- `VC` 凭证保管
- 签发方校验
- 凭证详情查看
- 选择性披露
- 证明请求处理
- 证明历史记录

设计目标：

- 让“可证明而不裸奔”成为用户可感知的体验

### 4.3 安全与恢复层

核心能力：

- `MPC` 无助记词架构
- 设备安全状态
- 恢复准备状态
- 新设备恢复流程
- 风险与责任说明

设计目标：

- 让用户感受到这是一种“比传统私钥钱包更容易理解、更可恢复”的安全模型

### 4.4 资产与支付层

核心能力：

- 资产概览
- 收付款
- 稳定币优先支付
- 自动 Gas 兑换
- 交易确认
- 交易回执

设计目标：

- 让支付体验接近 Web2，而不是逼用户理解原生代币逻辑

### 4.5 企业与组织层

核心能力：

- `E-Company` 创建与接入
- 企业档案
- 多签治理
- 企业金库
- 审批流
- 收入分润与结算记录

设计目标：

- 让钱包具备面向机构使用的严肃工作台能力

### 4.6 物理卡与线下扩展层

核心能力：

- 物理卡绑定
- 卡片状态
- `NFC` 交互
- 冻结、替换、重绑定
- 线下身份验证与支付场景

设计目标：

- 把数字身份延伸到物理世界

### 4.7 安全通信层

核心能力：

- `DIDComm` 消息能力
- 可信发件人展示
- 凭证请求通知
- 官方通知
- 组织通知

设计目标：

- 让钱包不仅能“持有身份”，还能“进行可信交互”

### 4.8 治理与合规层

核心能力：

- 治理参与
- 投票
- 提案查看
- 关键合规状态展示
- 大额跨境交易的审计友好支持

设计目标：

- 让产品成为未来公民权威体系的一部分，而不是体系之外的工具

## 5. 建议的产品结构

### 5.1 一个 App，两个上下文

建议不要把个人钱包、企业钱包、消息系统、卡片系统拆成多个独立应用，而应采用：

- `Citizen Mode（公民模式）`
- `Organization Mode（组织模式）`

这样做的优势在于：

- 底层身份体系是一套
- 钱包、消息、卡片、安全中心可共享
- 用户可能频繁在个人与组织身份之间切换
- 有利于维持整个 Future Citizen 的统一叙事

### 5.2 首页作为统一中枢

首页不应该只是余额页，而应是“我的当前身份与操作状态总览”。

首页应聚合：

- 身份状态
- 资产摘要
- 凭证摘要
- 待处理请求
- 组织入口
- 官方通知或安全提醒

## 6. 建议的信息架构

### 6.1 一级导航建议

面向移动端，建议使用以下五个一级导航：

- `Home`
- `Identity`
- `Pay`
- `Organization`
- `Inbox`

### 6.2 二级模块建议

在一级导航之下，补充以下核心模块：

- `Credentials`
- `Card`
- `Security`
- `Governance`
- `Settings`

### 6.3 为什么这样分

`Home`：
- 统一展示当前状态与关键动作入口

`Identity`：
- 对应产品真正的核心重心

`Pay`：
- 对应高频行为，必须足够直接

`Organization`：
- 给企业和组织场景独立而严肃的空间

`Inbox`：
- 承接 `DIDComm`、证明请求、官方通知与多签提醒

## 7. 核心页面设计框架

以下页面建议作为第一轮设计方案的主干。

### 7.1 Onboarding（引导与开户）

目标：

- 完成钱包创建、身份绑定与安全初始化

建议包含：

- 产品价值说明
- 创建钱包
- 绑定或认领 `FCDID`
- 绑定 `Digital ID`
- 设置基础安全方式
- 了解恢复机制
- 完成进入首页

设计重点：

- 降低技术恐惧感
- 不要让用户一上来就接触复杂链术语

### 7.2 Home Dashboard（首页总览）

目标：

- 让用户一眼看到“我是谁、我拥有什么、我该处理什么”

建议包含：

- 身份卡片
- 资产摘要
- 凭证摘要
- 快捷支付入口
- 待处理请求
- 组织切换入口
- 官方或安全消息

设计重点：

- 优先展示身份和信任状态
- 资产模块作为重要组成部分，但不是唯一视觉中心

### 7.3 Identity Center（身份中心）

目标：

- 集中管理个人数字身份能力

建议包含：

- `FCDID`
- `FNS` 名称
- 绑定国家 Digital ID 的状态
- 身份可信等级或状态说明
- 使用该身份可完成的事项

设计重点：

- 让身份信息简洁、正式、可信

### 7.4 Credential Vault（凭证中心）

目标：

- 管理所有 `VC` 与证明记录

建议包含：

- 全部凭证
- 签发方信息
- 有效状态
- 即将过期提醒
- 证明请求历史
- 选择性披露入口

设计重点：

- 让复杂的凭证逻辑变得直观
- 披露前一定要有可读的确认界面

### 7.5 Pay（支付与交易）

目标：

- 完成日常资金流转与无感支付

建议包含：

- 发送
- 接收
- 请求付款
- 稳定币支付
- 自动 Gas 说明
- 最近交易记录

设计重点：

- 默认隐藏链的复杂度
- 但交易完成后要有透明回执

### 7.6 Transaction Receipt（交易回执）

目标：

- 给用户一个可信、可审计、可复盘的结果页面

建议包含：

- 支付结果
- 支付对象
- 结算资产
- 自动 Gas 扣费说明
- 路由摘要
- 时间与交易 ID
- 风险与合规提示

设计重点：

- 结果要简单
- 细节要可展开

### 7.7 Security Center（安全与恢复中心）

目标：

- 让用户理解并管理自己的安全状态

建议包含：

- 当前设备状态
- 恢复是否已就绪
- 已绑定的身份信息
- 恢复方式说明
- 风险提示
- 卡片关联状态

设计重点：

- 安全说明必须清晰、可理解、不过度承诺

### 7.8 Organization Workspace（组织工作台）

目标：

- 支持数字企业与组织场景

建议包含：

- 企业资料
- 组织资产
- 签署人列表
- 当前权限结构
- 待审批事项
- 收入与分润记录

设计重点：

- 必须有明显“组织身份”标识
- 避免与个人资产混淆

### 7.9 Multisig Flow（多签审批流）

目标：

- 完成企业关键操作的多人授权

建议包含：

- 提案内容
- 发起人
- 当前签署进度
- 参与人状态
- 通过/拒绝操作
- 历史记录

设计重点：

- 责任边界清晰
- 每一步都要知道自己是以谁的身份在签

### 7.10 Card Center（实体卡中心）

目标：

- 管理物理 `ID Card`

建议包含：

- 卡片绑定
- 卡片状态
- `NFC` 使用说明
- 冻结与替换
- 离线场景能力说明

设计重点：

- 物理卡应该被设计成“可信扩展”，不是外设附属品

### 7.11 Inbox（可信消息中心）

目标：

- 承接所有可信交互与消息通知

建议包含：

- 官方通知
- 证明请求
- 企业审批提醒
- 服务消息
- 安全提醒

设计重点：

- 消息必须有可信来源标识
- 与普通聊天产品做出明确区分

### 7.12 Governance Center（治理中心）

目标：

- 为未来治理参与留出明确空间

建议包含：

- 提案列表
- 投票状态
- 部门分类
- 历史参与记录
- 身份绑定说明

设计重点：

- 治理界面要正式、稳定，不应被娱乐化

## 8. 建议的视觉方向

### 8.1 视觉关键词

建议视觉关键词如下：

- Institution-grade
- Future civic system
- Trusted infrastructure
- Calm power
- Structured modernity

### 8.2 推荐视觉语言

建议延续目前蓝图中的视觉母题，但做产品化转译：

- 深蓝作为系统背景基调
- 青蓝或冷光色作为结构信息强调
- 金色或浅金色作为权威、身份、核心节点强调
- 使用结构化线框、网格、层叠、证书感元素
- 避免过度炫技的链上发光效果

### 8.3 推荐气质

界面应让用户感到：

- 这是一个可信的数字制度系统
- 这是一个严肃但好用的公民工具
- 这不是一个高风险投机平台

## 9. 分阶段落地建议

### 9.1 第一阶段：基础公民钱包

建议优先落地：

- 引导与开户
- `FCDID`
- `FNS`
- `Digital ID` 绑定
- 基础资产页
- 支付与收款
- 自动 Gas
- 基础凭证中心
- 基础安全中心
- 基础消息中心

这一阶段的目标是：

- 先把“身份 + 钱包 + 信任入口”做成立

### 9.2 第二阶段：可信交互与企业能力

建议落地：

- 高级选择性披露
- 完整证明请求流
- `E-Company` 工作台
- 多签审批
- 收入分润可视化
- 更完整的 DIDComm 交互

这一阶段的目标是：

- 把产品从个人钱包升级为体系工作台

### 9.3 第三阶段：实体与治理扩展

建议落地：

- 物理卡中心
- 离线 `NFC` 交互
- 治理中心
- 合规协作视图
- 跨境与公共服务场景深化

这一阶段的目标是：

- 补齐 Future Citizen 的完整生态闭环

## 10. 当前建议的第一版设计重点

如果现在就要进入 UI/UX 设计，我建议第一版重点先做以下五个主模块：

- `Home`
- `Identity`
- `Credentials`
- `Pay`
- `Security`

同时准备两个扩展模块的框架稿：

- `Organization`
- `Inbox`

这样做的好处是：

- 能最快建立产品骨架
- 能先把最核心的身份逻辑设计扎实
- 为后续企业、消息、卡片、治理模块留下稳定扩展位

## 11. 目前需要继续确认的关键问题

在正式进入高保真设计前，建议继续明确以下问题：

- `MPC` 恢复机制到底是 `2-of-3` 还是 `3-of-3`
- 物理卡在恢复体系里究竟是额外分片、备份副本还是独立硬件凭证
- `FCDID`、`FCID`、`Future Citizen ID` 的命名边界是否统一
- 治理模块是否属于首发版本
- 首发核心用户到底优先面向个人公民、企业用户还是政府合作场景

## 12. 结论

综合目前已学习的所有材料，建议明确以下结论：

`FC Wallet` 不应被设计成“一个更高级的加密钱包”，而应被设计成 `Future Citizen` 体系的统一数字门户。

最合理的设计方向是：

- 以身份为中心
- 以信任为表达
- 以支付为高频入口
- 以组织能力为重要扩展
- 以消息与凭证为可信交互主线
- 以安全恢复为基础保障
- 以实体卡与治理能力作为后续战略延展

这套框架既能支撑第一阶段产品落地，也能为更长线的未来公民体系扩展保留完整空间。
